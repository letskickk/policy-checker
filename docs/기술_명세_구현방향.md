# 기술 명세 및 구현 방향

## 1. 확정 사항

| 항목 | 내용 |
|------|------|
| 데이터 제공 방식 | **PDF** (정강·정책·과거 공약) |
| 분석 엔진 | **GPT API** (OpenAI) |
| 1차 기능 | **당 부합 점검** (중복·유사 탐지는 2차) |

---

## 2. 데이터: PDF 처리

### 2.1 역할

- **입력**: 정강·정책·과거 공약 PDF 파일
- **처리**: PDF → 텍스트 추출 → GPT가 참조할 기준 문서로 사용

### 2.2 보관 구조

```
data/pdf/
├── 정강.pdf          # 당 정강 (예시)
├── 정책.pdf          # 당 정책/정책기조 (예시)
└── 과거공약.pdf      # 과거 공약 집약본 (예시)
```

- 파일명·개수는 실제 제공 분량에 맞춰 조정
- PDF 추가·교체 시 재파싱만 하면 되어 확장 용이

### 2.3 기술 선택 (권장)

- **Python**: `pypdf` 또는 `pdfplumber`로 텍스트 추출
- **대용량/복잡 레이아웃**: 필요 시 `PyMuPDF(fitz)` 검토
- 추출 텍스트는 **캐시**(파일/DB) 저장해, 요청마다 재파싱하지 않도록 구성

### 2.4 제약 대응

- **토큰 한도**: GPT 컨텍스트 길이 제한에 맞춰, 요약·청크 분할·핵심 문단만 넣기 등 필요
- **PDF 품질**: 스캔본/이미지 PDF는 OCR(예: Tesseract) 검토

---

## 3. 분석: GPT API

### 3.1 담당 역할

- **당 부합 점검**: 정강·정책·과거 공약(추출 텍스트)을 컨텍스트로 제공하고, 사용자 공약에 대해  
  **부합/부분부합/미부합** 판단 + **근거** + **수정·보완 체크리스트** 생성

### 3.2 호출 구조 (개념)

1. **컨텍스트 구성**
   - PDF에서 추출한 텍스트(또는 요약/청크)를 시스템 메시지 또는 사용자 메시지 앞단에 포함
2. **요청**
   - 사용자 메시지: “다음 공약을 당 방향에 맞게 점검해 주세요: [출마자 공약]”
3. **응답**
   - 구조화된 형태로 받기 위해 **프롬프트에서 지시**  
     (부합 여부, 근거, 체크리스트를 구분해 작성하도록)

### 3.3 모델·비용

- **모델**: GPT-4o 또는 GPT-4o-mini 등(속도·비용·품질 균형)
- **비용**: 컨텍스트 길이·요청 빈도에 따라 산정; 필요 시 캐시·요약으로 토큰 절감

### 3.4 보안

- **API 키**: 환경 변수 등으로 관리, 저장소에 커밋 금지
- **공약 내용**: 로깅·저장 정책 결정 후, 필요 시 API 전송 데이터 최소화

## 4. 당 부합 점검 플로우 (1차 MVP)

```
[정강·정책·과거공약 PDF]
         │
         ▼
   PDF 텍스트 추출 ──► 캐시 저장 (선택)
         │
         ▼
[출마자 공약 입력]
         │
         ▼
   컨텍스트 구성: 기준 문서 + "다음 공약을 점검해 주세요: ..."
         │
         ▼
   GPT API 호출
         │
         ▼
   응답 파싱 ──► 부합 여부 / 근거 / 수정·보완 체크리스트
         │
         ▼
   사용자에게 결과 반환
```

### 4.1 입력

- 출마자 공약 **텍스트** (한 건 또는 여러 항목 한 번에)

### 4.2 출력 (목표 형식)

- **종합**: 부합 / 부분부합 / 미부합
- **항목별** (공약이 여러 개인 경우): 항목별 부합 여부
- **근거**: 어떤 정강·정책·과거 공약 조항과 맞는지/어긋나는지
- **체크리스트**: 수정·보완이 필요한 항목을 짧은 체크리스트로 안내

---

## 5. 프로젝트 구성 제안

| 구성요소 | 내용 |
|----------|------|
| **data/pdf/** | PDF 원본 보관 |
| **backend** | PDF 파싱, 텍스트 캐시, GPT API 호출, 당 부합 점검 API(엔드포인트) |
| **prompts/** | 당 부합 점검용 시스템·유저 프롬프트 문구 |
| **frontend** | 2차에서 진행 가능 (우선 API만 구현해도 동작 가능) |

---

## 6. 다음 구현 단계

1. **data/pdf/** 에 샘플 또는 실제 PDF 배치
2. **PDF 파싱 스크립트** 작성 → 추출 텍스트 확인
3. **prompts** 에 당 부합 점검용 프롬프트 초안 작성
4. **GPT API** 호출 코드 작성 (컨텍스트 = 추출 텍스트, 질의 = 공약)
5. **응답 형식** 고정(예: JSON) 후 파싱·체크리스트 표시 로직
6. **API 서버** (FastAPI 등)로 엔드포인트 제공 → 이후 프론트/챗 연동
